Patrón Arquitectónico

MVC por capas con separación clara: Rutas → Controladores → Servicios → Helpers. Patrón ahora uniforme en todos los módulos — incluyendo correo y notificaciones que fueron refactorizados.

Módulos del Proyecto

  • server.js — Bootstrap, middleware, rate limiters
  • routes/ — Routers Express, wiring puro
  • controllers/ — Manejo de petición/respuesta HTTP
  • services/ — Lógica de negocio y consultas BD
  • middleware/ — Auth (JWT + Google OAuth)
  • helpers/ — Pool BD, S3, mailer, logger, upload
  • db/ — Schema, migraciones, seed

Dependencias Externas

  • MySQL2 — almacenamiento relacional
  • AWS S3 (compatible) — almacenamiento de documentos
  • Resend — correo transaccional
  • Google OAuth2 — inicio de sesión SSO
  • JWT + bcryptjs — autenticación y hashing
  • Winston — logging estructurado
  • Helmet + express-rate-limit — seguridad HTTP

Modelo de Autorización

  • admin — acceso total a todos los recursos
  • usuario — acotado a su obra_id asignada
  • JWT lleva id, rol, obra_id, jti
  • Revocación de tokens vía tabla sesiones_revocadas (lista negra JTI)
Hallazgos de Arquitectura
Resuelto Lógica de Negocio Filtrada Hacia los Route Handlers

Corregido: Se crearon services/notificacionesService.js, controllers/correoController.js y controllers/notificacionesController.js. La lógica de BD se movió a los servicios, y routes/correo.js y routes/notificaciones.js quedaron en wiring puro (~13 líneas cada uno). La arquitectura Routes → Controllers → Services es ahora uniforme en todo el proyecto.

Medio Sin Middleware de Validación de Entradas

No existe una librería de validación estructurada (Joi, Zod, express-validator). La validación de entradas es ad-hoc — solo verificaciones de campos requeridos. Restricciones como longitud máxima, formato de correo, valores de enum y formato UUID no se aplican en la capa API.

Por ejemplo, el campo rol en POST /api/usuarios se pasa directamente a la BD. Si se envía un rol inválido, el ENUM de MySQL lo rechazará con un error crudo en lugar de una respuesta 400 limpia.

Afecta: Todos los controladores — controllers/usuariosController.js controllers/obrasController.js etc.

Resuelto Sin Paginación en Endpoints de Listado

Corregido: Los tres endpoints de listado ahora soportan ?page y ?limit (default: 20, máximo: 100). La respuesta incluye el objeto pagination: { total, page, limit, pages }. Los servicios ejecutan un COUNT(*) seguido de SELECT ... LIMIT ? OFFSET ?.

Archivos actualizados: services/obrasService.js · services/usuariosService.js · services/documentosService.js y sus controladores correspondientes.

Bajo Estrategia de Eliminación Inconsistente

Los usuarios y obras se eliminan de forma lógica (activo = 0), pero los proyectos y documentos se eliminan físicamente. No existe trazabilidad de los registros eliminados físicamente. Si un proyecto o documento se borra por error, no hay forma de recuperarlo.

Se recomienda unificar la estrategia de borrado lógico o implementar una tabla de auditoría.

Afecta: services/proyectosService.js:44   services/documentosService.js:82

Bajo Sin Versionado de API

Todas las rutas están montadas bajo /api/... sin prefijo de versión (ej. /api/v1/...). Cualquier cambio que rompa compatibilidad afectará a todos los consumidores simultáneamente sin ruta de migración.

Info Helper de BD Importado Inline Dentro de una Función

En documentosController.js línea 78, el pool de base de datos se requiere dentro del cuerpo de una función (require('../helpers/db')) en lugar de en el nivel superior del módulo. Aunque Node.js cachea módulos, esto es inconsistente y hace poco claras las dependencias.

Afecta: controllers/documentosController.js:78

Fortalezas de Seguridad
✓
Cabeceras de seguridad con Helmet.js — Aplica automáticamente X-Content-Type-Options, X-Frame-Options, HSTS, CSP y otras cabeceras HTTP protectoras.
✓
Lista blanca estricta de orígenes CORS — Solo se permiten FRONTEND_URL y EXTRA_ORIGINS. Localhost solo en entornos no productivos. Sin comodines. Orígenes bloqueados se registran con mensaje descriptivo.
✓
Rate limiting en endpoints sensibles — Login: 10 req/15min por IP. Recuperar contraseña: 5 req/hr por IP. Correo: 10 req/hr por IP.
✓
Revocación de tokens JWT (lista negra JTI) — Cada token lleva un UUID jti. Al hacer logout, se inserta en sesiones_revocadas. El middleware valida esta tabla en cada petición.
✓
bcrypt para hashing de contraseñas — Factor de costo 10. Las contraseñas nunca se almacenan en texto plano ni se devuelven en respuestas de la API.
✓
Prevención de inyección SQL — Todas las consultas usan mysql2 con placeholders ? parametrizados. Sin interpolación de cadenas en SQL.
✓
Validación de tipo de archivo con magic bytes — Uploads validan MIME type contra lista blanca y verifican las firmas de bytes reales del archivo (PDF, PNG, JPEG, etc.) antes de subir a S3. Corregido: ahora se llama correctamente en el flujo de subida.
✓
JWT con expiración garantizada — Tokens firmados con process.env.JWT_EXPIRES_IN || '8h'. Si la variable de entorno no está configurada, se aplica un límite de 8 horas por defecto. Corregido en esta iteración.
✓
Control de acceso basado en roles (RBAC) — requireRole('admin') protege todos los endpoints exclusivos de administrador. El rol está embebido en el JWT verificado.
✓
Acotación de recursos por obra — requireObraAccess garantiza que usuarios no administradores solo accedan a datos de su obra_id, previniendo escalada horizontal de privilegios.
✓
Sanitización de errores en producción — El manejador global devuelve 'Error interno del servidor' en producción. Sin stack traces expuestos al cliente.
✓
Respuesta genérica en recuperar contraseña — El flujo devuelve el mismo mensaje independientemente de si el correo existe, previniendo enumeración de usuarios.
✓
TLS en la base de datos de producción — Conexión a BD con SSL y rejectUnauthorized: true por defecto en entornos no-desarrollo.
✓
Auto-eliminación prevenida — Un administrador no puede eliminar su propia cuenta (usuariosController.js:46).
✓
Escape de HTML en plantillas de correo — Valores del usuario renderizados en HTML de correo procesados con escapeHtml(), previniendo inyección HTML.
Hallazgos de Seguridad
Resuelto La Validación de Magic Bytes Estaba Definida Pero Nunca Se Llamaba

Corregido: Se importa validateMagicBytes desde helpers/upload.js en services/documentosService.js y se llama por cada archivo dentro de insertMany() antes de subir a S3. Si el contenido del archivo no coincide con el MIME type declarado, se lanza un error 400 con mensaje descriptivo y la subida se cancela.

Medio Sin Protección de Fuerza Bruta por Cuenta

El rate limiting se aplica únicamente por dirección IP (10 intentos / 15 min). Un atacante con múltiples IPs puede realizar ataques de fuerza bruta distribuidos contra una cuenta específica sin alcanzar el límite por IP.

No existe bloqueo de cuenta, contador de intentos fallidos por usuario, ni desafío CAPTCHA tras fallos repetidos.

Solución recomendada: Rastrear intentos fallidos por email en BD. Tras N fallos (ej. 10), bloquear la cuenta temporalmente o requerir confirmación por correo para desbloquear.

Afecta: services/authService.js:35   server.js:75

Resuelto JWT_EXPIRES_IN Sin Valor por Defecto

Corregido: La firma del token ahora usa process.env.JWT_EXPIRES_IN || '8h'. Si la variable de entorno no está configurada, los tokens expiran en 8 horas por defecto, eliminando el riesgo de tokens sin expiración.

Bajo confirmUpload No Valida el Tipo MIME

El endpoint POST /:obraId/confirm-upload valida que key empiece con documentos/, pero no verifica que mimetype esté en la lista ALLOWED_MIME. Cualquier cadena de tipo MIME arbitraria puede almacenarse en la base de datos.

Afecta: controllers/documentosController.js:66

Bajo La Tabla sesiones_revocadas No Tiene Mecanismo de Limpieza

La tabla almacena JTIs revocados con columna expira_en, pero ningún proceso elimina filas expiradas. A largo plazo, la tabla crecerá indefinidamente, aumentando el tiempo de consulta en cada petición autenticada.

Solución: Evento de BD programado o cron de Node.js para ejecutar DELETE FROM sesiones_revocadas WHERE expira_en < NOW().

Bajo Longitud Mínima de Contraseña Solo Aplicada en Reset, No en Creación

resetPassword verifica password.length < 6. Sin embargo, POST /api/usuarios (crear usuario) no tiene verificación de longitud. Un administrador podría crear un usuario con una contraseña de 1 carácter.

Afecta: controllers/authController.js:69 (tiene la verificación) vs controllers/usuariosController.js:20 (sin verificación)

Bajo El Campo rol No Se Valida Contra el Enum en la Capa de Aplicación

El valor de rol del body se pasa directamente al servicio sin validación en la capa de aplicación. Si se envía un valor inválido, el ENUM de MySQL lanzará un error crudo en vez de un 400 descriptivo.

Solución: if (!['admin','usuario'].includes(rol)) return res.status(400)...

Afecta: controllers/usuariosController.js:27

Info Correos Electrónicos Registrados a Nivel Info

Los intentos de login registran el correo completo a nivel info. Los archivos de log en logs/app.log podrían exponer PII si se envían a un agregador externo sin enmascarar.

Solución: Enmascarar correos en logs (ej. u***@dominio.com) o bajar a nivel debug.

Afecta: services/authService.js:37

Corregido: El nuevo controllers/correoController.js usa logger.error (Winston) en todos los handlers. El uso de console.error en routes/correo.js fue eliminado junto con la lógica que contenía.
